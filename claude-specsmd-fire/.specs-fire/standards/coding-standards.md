# Coding Standards

## Overview

Código limpio, simple y extensible. Separación clara de capas en backend y organización por
feature en frontend. Sin sobreingeniería: el MVP prioriza claridad y que cada feature funcione.

## Naming Conventions

### Backend (Python)

| Element | Convention | Example |
|---------|------------|---------|
| Módulos/archivos | snake_case | `friction_service.py` |
| Clases | PascalCase | `FrictionService` |
| Funciones/variables | snake_case | `calculate_monthly_hours` |
| Constantes | UPPER_SNAKE | `DEFAULT_HOURLY_RATE` |
| Campos API (JSON) | camelCase | `timeLostMinutes` |

> Los modelos de dominio usan camelCase en el JSON expuesto (alias) para coincidir con el
> contrato del frontend Angular, manteniendo snake_case internamente cuando aplique.

### Frontend (TypeScript/Angular)

| Element | Convention | Example |
|---------|------------|---------|
| Componentes | kebab-case archivo / PascalCase clase | `friction-list.component.ts` |
| Servicios | kebab-case archivo / PascalCase clase | `friction.service.ts` |
| Interfaces/Modelos | PascalCase | `Friction`, `Initiative` |
| Variables/funciones | camelCase | `loadFrictions()` |

## File Organization

### Backend

```
backend/app/
├── main.py              # App FastAPI, CORS, routers, startup (seed)
├── core/                # config, database (engine/session)
├── models/              # Entidades SQLModel (tablas)
├── schemas/             # DTOs de entrada/salida (Pydantic)
├── repositories/        # Acceso a datos
├── services/            # Lógica: impacto, clasificación, priorización, CRUD
└── api/                 # Routers por recurso (frictions, initiatives, dashboard)
```

### Frontend

```
frontend/friccion-cero-web/src/app/
├── core/                # modelos, servicios HTTP, enums
├── features/
│   ├── dashboard/
│   ├── frictions/
│   └── initiatives/
└── shared/              # componentes/utilidades reutilizables
```

## Error Handling

### Backend

- Validación con Pydantic/SQLModel; errores 422 automáticos.
- Recursos inexistentes → `HTTPException(404)`.
- Lógica de negocio inválida → `HTTPException(400)` con mensaje claro.

### Frontend

- Servicios HTTP devuelven Observables; manejar errores con `catchError` y mostrar `MessageService` (toast) de PrimeNG.

## Comments and Documentation

- Comentar el *por qué*, no el *qué*.
- Docstrings cortos en servicios con lógica no obvia (fórmulas de impacto, reglas de priorización).
- Endpoints documentados vía FastAPI (summary/response_model) → aparecen en Swagger.

## Anti-Patterns to Avoid

- **Datos mock como fuente principal**: el frontend final consume la API real.
- **Lógica de negocio en controladores/componentes**: vive en services.
- **Sobreingeniería**: nada de capas o abstracciones que el MVP no necesita.
