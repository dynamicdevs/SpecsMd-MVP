# Testing Standards

## Overview

Para el MVP el foco está en que cada feature funcione end-to-end. Se priorizan pruebas de
la lógica de negocio pura (cálculo de impacto, clasificación por reglas, priorización) por ser
determinista y de alto valor, y verificación manual de los endpoints vía Swagger.

## Testing Framework

**Framework**: pytest (backend) / Karma+Jasmine o Vitest (frontend, según genere Angular CLI)
**Runner**: pytest / ng test

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Unit | pytest | `backend/tests/` | Lógica de impacto, clasificación, priorización |
| API smoke | Swagger / httpx | `/docs` | Verificar endpoints manualmente en el MVP |
| Component | Angular testing | `*.spec.ts` | Componentes clave (opcional en MVP) |

## Coverage Requirements

**Target**: Lógica de negocio del backend (services) con cobertura prioritaria.
**Enforcement**: No bloqueante en el MVP.

**Critical paths that MUST have coverage:**
- `MonthlyHoursLost = TimeLostMinutes * Frequency * PeopleAffected / 60`
- `EstimatedMonthlyCost = MonthlyHoursLost * HourlyRate`
- Reglas de clasificación por texto
- Reglas de priorización (Crítica/Alta/Media/Baja)

## Test Naming

**Pattern**: `test_<unidad>_<condicion>_<resultado_esperado>`

**Examples**:
- `test_monthly_hours_basic_calculation` — verifica la fórmula de horas.
- `test_classify_contains_excel_returns_trabajo_manual` — regla de clasificación.
- `test_priority_high_cost_high_pain_returns_critica` — regla de priorización.

## Running Tests

```bash
# Backend
cd friccion-cero/backend
pytest

# Frontend
cd friccion-cero/frontend/friccion-cero-web
ng test
```
