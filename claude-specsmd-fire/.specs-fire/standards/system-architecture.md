# System Architecture

## Overview

Fricción Cero es una aplicación cliente-servidor: una SPA Angular consume una API REST de FastAPI
que persiste en SQLite. La lógica de impacto, clasificación y priorización vive en el backend.

## System Context

Equipos/empresas registran fricciones operativas; el sistema calcula su impacto (horas y costo),
las clasifica y prioriza, y permite convertirlas en iniciativas de mejora con seguimiento.

### Context Diagram

```
[Usuario] --> [Angular SPA + PrimeNG] --HTTP/JSON--> [FastAPI API] --> [SQLite]
```

### Users

- **Equipo / Empresa**: registra y gestiona fricciones e iniciativas, consulta el dashboard.

### External Systems

- Ninguno en el MVP (sin IA externa; clasificación por reglas locales).

## Architecture Pattern

**Pattern**: API REST en capas (Router → Service → Repository → Model) + SPA por features.
**Rationale**: Simple, modular y extensible; cumple la separación pedida (Controllers/Services/
Repositories/DTOs/Entities) trasladada al ecosistema Python.

## Component Architecture

### Components

#### API (FastAPI)
- **Purpose**: Exponer endpoints REST y documentación OpenAPI.
- **Responsibilities**: routing, validación de entrada/salida (schemas), orquestar services.
- **Dependencies**: services, schemas.

#### Services
- **Purpose**: Lógica de negocio.
- **Responsibilities**: cálculo de impacto, clasificación por reglas, priorización, CRUD.
- **Dependencies**: repositories, models.

#### Repositories
- **Purpose**: Acceso a datos.
- **Responsibilities**: queries y persistencia sobre SQLModel/SQLite.
- **Dependencies**: models, session.

#### Frontend (Angular + PrimeNG)
- **Purpose**: UI por features.
- **Responsibilities**: formularios reactivos, servicios HTTP por feature, pantallas, dashboard.
- **Dependencies**: API REST.

### Component Diagram

```
api/ (routers) -> services/ -> repositories/ -> models/ (SQLModel) -> SQLite
schemas/ (DTOs) <-> api/
frontend features/ -> core/services (HttpClient) -> API
```

## Data Flow

El frontend envía/recibe DTOs JSON (camelCase). Al crear/editar una fricción, el backend recalcula
`MonthlyHoursLost`, `EstimatedMonthlyCost` y `Priority`; opcionalmente sugiere `Category` por reglas
de texto. El dashboard agrega métricas leyendo el repositorio.

## Technology Stack

| Layer | Technology | Purpose |
|-------|------------|---------|
| Frontend | Angular + PrimeNG | UI por features, formularios reactivos |
| API | FastAPI | REST + OpenAPI/Swagger |
| Lógica | Python services | impacto, clasificación, priorización |
| Datos | SQLModel | ORM/entidades |
| DB | SQLite | persistencia local |

## Non-Functional Requirements

### Security
- CORS restringido al origen del frontend en desarrollo.
- Sin secretos en código; valor hora configurable por variable de entorno.

### Scalability
- SQLite es suficiente para el MVP; migrable a Postgres a futuro cambiando la URL del engine.

## Constraints

- No usar IA externa (clasificación por reglas simples).
- No dejar datos mock como fuente principal.
- No sobreingenierizar.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Backend stack | FastAPI + SQLModel (en lugar de .NET 8) | Petición explícita del usuario de usar otra tecnología; además .NET SDK no está instalado, Python 3.12 sí. |
| Frontend stack | Angular + PrimeNG | Mantiene el requisito original de UI. |
| Estructura | Monorepo `friccion-cero/{backend,frontend}` | Simplicidad para MVP fullstack; las 4 capas .NET se mapean a routers/services/repositories/models+schemas. |
| Migraciones | `create_all` de SQLModel (Alembic futuro) | MVP: menos fricción; el esquema es estable y pequeño. |
| Cálculo de impacto/priorización | En backend (services) | Single source of truth; el frontend solo muestra. |
