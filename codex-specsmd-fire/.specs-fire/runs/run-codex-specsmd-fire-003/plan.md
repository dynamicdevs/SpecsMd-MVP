# Implementation Plan: Domain Model and Persistence

## Approach

Add the initial domain and persistence layer for Friccion Cero using Prisma + SQLite, TypeScript domain types, Zod DTO schemas, and repository interfaces/implementations.

This work item will create the data model and repository foundation only. It will not expose CRUD HTTP endpoints yet; those belong to the `friction-api` and `initiative-api` work items.

Planned model relationships:

```text
Friction
  1 -> many FrictionComment
  1 -> many Initiative

Initiative
  many -> 1 Friction

FrictionComment
  many -> 1 Friction
```

Use simple string-backed enums in Prisma and TypeScript for MVP flexibility:

- FrictionCategory
- FrictionStatus
- Frequency
- PainLevel
- AutomationPotential
- Priority
- InitiativeComplexity
- InitiativeStatus

## Technical Details

### Prisma Fields

`Friction`:

- `id`
- `title`
- `description`
- `area`
- `category`
- `frequency`
- `timeLostMinutes`
- `peopleAffected`
- `painLevel`
- `automationPotential`
- `monthlyHoursLost`
- `estimatedMonthlyCost`
- `priority`
- `status`
- `createdAt`
- `updatedAt`

`Initiative`:

- `id`
- `frictionId`
- `title`
- `proposedSolution`
- `expectedReductionPercent`
- `complexity`
- `priority`
- `status`
- `createdAt`
- `updatedAt`

`FrictionComment`:

- `id`
- `frictionId`
- `comment`
- `createdAt`

### Practical Choices

- Use `String` IDs generated by `cuid()` for API-friendly identifiers.
- Store category/status/priority/frequency as strings for simple MVP evolution.
- Use Prisma `Decimal` or `Float` for `monthlyHoursLost` and `estimatedMonthlyCost`; prefer `Float` for MVP simplicity.
- Use `prisma db push` instead of formal migrations during the earliest MVP iteration. A later hardening pass can introduce named migrations.
- Repository interfaces stay small and shaped around near-term CRUD/API needs.
