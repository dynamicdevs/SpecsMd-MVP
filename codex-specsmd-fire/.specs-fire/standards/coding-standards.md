# Coding Standards

## Overview

This project is primarily a skill and workflow package. Keep files readable, deterministic, and easy for agents to consume.

## Code Formatting

**Tool**: Repository/editor defaults  
**Config**: Not configured  
**Enforcement**: Manual review until automated tooling is added

### Key Settings

- **Encoding**: UTF-8
- **Line endings**: Preserve repository defaults
- **Indentation**: Two spaces for YAML and Markdown examples

## Linting

**Tool**: Manual review  
**Base Config**: Not configured  
**Strictness**: Prefer simple, explicit files over clever formatting

### Key Rules

- `skill-frontmatter`: Each skill file should keep valid YAML frontmatter.
- `stable-paths`: Referenced paths must exist or be clearly marked as install targets.
- `script-scope`: Helper scripts should only mutate intended FIRE artifacts.

| Element | Convention | Example |

## Files and Folders

- **Skills**: Place Codex skills under `.codex/skills/{skill-name}/SKILL.md`.
- **Agents**: Place FIRE agents under `.specsmd/fire/agents/{agent-name}/agent.md`.
- **Skill helpers**: Place scripts/templates under the owning skill directory.
- **Generated FIRE artifacts**: Place project state, intents, runs, and standards under `.specs-fire/`.

## Import Order

```javascript
const fs = require("node:fs");
const path = require("node:path");
```

**Rules**:

- Prefer Node built-in modules with `node:` prefixes.
- Keep script dependencies minimal unless a package manifest is introduced.
- Parse structured data with a parser when practical.

## Error Handling

### Pattern

**Approach**: Fail with clear messages and non-zero exits for invalid state.

### Guidelines

- Validate required paths before writing.
- Include the artifact path in error messages.
- Avoid silent state changes.

### Example

```javascript
if (!fs.existsSync(statePath)) {
  console.error(`Missing FIRE state: ${statePath}`);
  process.exit(1);
}
```

## Logging

**Tool**: Console output for helper scripts  
**Format**: Short, action-oriented messages

### Log Levels

| Level | Usage |
|-------|-------|
| Info | Successful state transitions and artifact creation |
| Warn | Recoverable missing optional files |
| Error | Missing required state or invalid input |

### Guidelines

**Always log**:

- Created run IDs or updated run phases
- Missing required state files
- Completion status changes

**Never log**:

- Secrets or credentials
- Full external untrusted payloads
- User-private content unrelated to the FIRE artifact being updated

## Comments and Documentation

### When to Comment

- Explain non-obvious state transitions.
- Document schema changes in the relevant standards or skill file.
- Keep comments close to the behavior they clarify.

### Documentation Format

**Functions**: Short comments only when behavior is not obvious  
**Classes**: Not currently used

## Code Patterns

### Preferred Patterns

#### Explicit Path Construction

Resolve paths from the package or workspace root and keep mutations scoped.

```javascript
const runPath = path.join(root, ".specs-fire", "runs", runId);
```

### Anti-Patterns to Avoid

- **Implicit global state**: Makes agent behavior difficult to audit.
- **Unverified path writes**: Can mutate the wrong workspace.
- **Stale command docs**: Breaks handoff between orchestrator, planner, and builder.
