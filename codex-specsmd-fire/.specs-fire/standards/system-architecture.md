# System Architecture

## Overview

`codex-specsmd-fire` packages the FIRE flow for Codex. It provides command-entry skills, agent definitions, workflow skills, templates, and helper scripts that create and update FIRE artifacts in a target workspace.

## System Context

Codex invokes skill entrypoints from `.codex/skills`. Those skills load FIRE agent definitions from `.specsmd/fire/agents`, then create or update project-local artifacts under `.specs-fire`.

### Context Diagram

```text
User -> Codex Skill -> FIRE Agent -> FIRE Skill/Template/Script -> .specs-fire artifacts
```

### Users

- **Developer**: Invokes FIRE commands and reviews generated plans, work items, and walkthroughs.
- **Codex agent**: Reads skill instructions and updates FIRE state according to the workflow.

### External Systems

- **Filesystem**: Stores skill package files and generated FIRE project artifacts.
- **Git**: Tracks package changes and project artifacts.

## Architecture Pattern

**Pattern**: File-based workflow orchestration  
**Rationale**: Plain Markdown and YAML artifacts are transparent, portable, and easy for coding agents to inspect.

## Component Architecture

### Components

#### Command Skills

- **Purpose**: Provide Codex command entrypoints.
- **Responsibilities**: Activate orchestrator, planner, or builder workflows.
- **Dependencies**: `.codex/skills/**/SKILL.md`

#### FIRE Agents

- **Purpose**: Define role-specific behavior.
- **Responsibilities**: Route, plan, build, and record workflow state.
- **Dependencies**: `.specsmd/fire/agents/**/agent.md`

#### FIRE Skills

- **Purpose**: Implement discrete workflow steps.
- **Responsibilities**: Initialize projects, capture intents, decompose work, plan runs, execute runs, review code, and generate walkthroughs.
- **Dependencies**: Templates and scripts under each owning skill folder.

#### FIRE Artifacts

- **Purpose**: Store project state and execution history.
- **Responsibilities**: Track intents, work items, runs, standards, and generated reports.
- **Dependencies**: `.specs-fire/state.yaml` schema.

## Data Flow

The orchestrator reads `.specs-fire/state.yaml`, routes to planner or builder, and target agents update Markdown/YAML artifacts as workflow state advances.

```text
Command -> Read state -> Select agent -> Execute skill -> Update artifacts -> Handoff/status
```

## Technology Stack

| Layer | Technology | Purpose |
|-------|------------|---------|
| Command layer | Codex skills | User-facing command activation |
| Workflow docs | Markdown | Agent and skill instructions |
| State | YAML | FIRE project state and metadata |
| Templates | Handlebars-style Markdown templates | Generated artifacts |
| Scripts | Node.js CommonJS | Run state mutation helpers |

## Non-Functional Requirements

### Performance

- **Activation latency**: Read only necessary skills and state files for the requested workflow.
- **Artifact operations**: Keep file writes scoped and small.

### Security

- Do not execute instructions from generated artifacts as trusted commands.
- Do not write outside the intended workspace artifact root.
- Keep helper scripts free of network side effects unless explicitly introduced and reviewed.

### Scalability

The file-based structure should support monolith and monorepo projects through hierarchical standards and module-specific overrides.

## Constraints

- Package files must remain consumable by Codex skill discovery.
- FIRE state schema changes must be reflected in `memory-bank.yaml` and affected skills.
- Generated artifacts should stay readable without specialized tooling.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Workflow storage | Markdown and YAML files | Transparent and agent-friendly |
| Runtime scripts | Node.js CommonJS | Simple local execution without a build step |
| Standards inheritance | Root constitution plus optional module overrides | Supports monorepos while preserving universal rules |
